# HVV Traffic Analyzer — 项目手册

> 护网行动实时流量分析工具 | 版本：v1.0 | 日期：2026-06-19

---

## 一、项目简介

本工具是面向护网行动（HVV）蓝队的实时流量分析脚本，基于 tshark 抓包引擎，实现三分类流量分析：

| 分类 | 处理方式 | 终端颜色 |
|------|---------|---------|
| **确认攻击** | 实时告警 + 统计报告 | 🔴 红色 |
| **可疑流量** | 实时提示 + 导出文本文件（含 Wireshark 检索词） | 🟡 黄色 |
| **普通流量** | 仅计数，不输出 | — |

**系统依赖**：Python 3.10+、Wireshark（含 tshark，需在 PATH 中）

## 三、文件详解

### 3.1 `main.py` — 程序入口（281 行）

**职责**：解析命令行参数、组装所有模块、运行主循环、优雅退出

**流程**：
1. 解析 CLI 参数（argparse）
2. 初始化：过滤器 → 检测引擎 → IOC 提取器 → 可疑检测引擎 → 导出器
3. 启动 tshark 抓包
4. 主循环：逐包处理 → 过滤 → 攻击检测 → 可疑检测 → IOC 提取 → 统计
5. Ctrl+C 退出时：关闭抓包 → 输出报告 → 导出文件

**可修改内容**：
- `parse_args()` 函数：新增 CLI 参数
- `build_filter()` 函数：自定义过滤规则构建逻辑
- 主循环中的模块调用顺序

---

### 3.2 `core/capture.py` — tshark 抓封装（148 行）

**职责**：封装 tshark 子进程，提供流式数据包读取

**类**：
- `TsharkCapture` — 管理 tshark 进程生命周期
  - `start()` — 启动 tshark（`-T ek` Elastic JSON 输出）
  - `stop()` — 优雅终止（terminate → wait → kill）
  - `packets()` — 逐包 yield dict（阻塞式读取）
  - `list_interfaces()` — 调用 `tshark -D` 列出网卡

**关键实现**：
- 使用 `subprocess.Popen` 以行缓冲模式启动
- `-T ek` 格式每包输出两行 JSON（index 行 + 数据行），跳过 index 行
- 启动提示行（非 JSON）自动跳过

**一般不需要修改**，除非要换抓包引擎（如 scapy）。

---

### 3.3 `core/parser.py` — 数据包解析（305 行）

**职责**：将 tshark EK JSON 解析为统一的 `Packet` 结构体

**数据类**：
```
Packet
├── 基础字段：timestamp, src_ip, dst_ip, src_port, dst_port, protocol, length
├── http: HTTPInfo (method, uri, host, user_agent, content_type, cookie, body)
├── dns:  DNSInfo  (query_name, query_type, response, is_response)
└── tls:  TLSInfo  (version, sni, ja3, ja3s, ja4, cert_subject, cert_issuer)
```

**关键函数**：
- `parse_packet(raw: dict) -> Packet` — 主解析函数
- `_parse_http()`, `_parse_dns()`, `_parse_tls()` — 子协议解析

**一般不需要修改**，除非 tshark 输出字段变化或要支持新协议。

---

### 3.4 `core/filter.py` — 过滤引擎（192 行）

**职责**：根据 CLI 参数构建过滤规则，筛选目标流量

**类**：
- `FilterRule` — 单条过滤规则（dataclass）
  - 五元组：src_ips, dst_ips, src_ports, dst_ports, protocols
  - 内容：content_patterns（正则）
  - HTTP：http_methods, http_uri_pattern, http_ua_pattern, http_content_type
  - DNS：dns_domain_pattern, dns_types
- `TrafficFilter` — 过滤引擎
  - `add_rule()` — 添加规则（自动预编译正则）
  - `match(packet)` — 返回命中的规则列表

**匹配逻辑**：单条规则内所有条件 AND，多条规则之间 OR

**一般不需要修改**。

---

### 3.5 `core/detector.py` — 确认攻击检测（312 行）

**职责**：基于 `rules/signatures.yaml` 检测确认攻击行为

**类**：
- `Alert` — 告警数据类（rule_id, rule_name, category, severity, packet, matched_detail）
- `AttackDetector` — 检测引擎
  - `_load_rules(path)` — 加载 YAML 规则并预编译正则
  - `enable_categories(categories)` — 选择性启用类别
  - `detect(packet) -> list[Alert]` — 执行检测

**两种检测模式**：
- `mode: pattern` — 正则模式匹配（target_fields + patterns）
- `mode: condition` — 复合条件匹配（conditions 列表，所有条件 AND）

**支持的条件操作符**：
| 操作符 | 说明 | 示例 |
|--------|------|------|
| `eq` | 等于 | `method == "POST"` |
| `contains` | 包含 | `content_type contains "octet-stream"` |
| `regex` | 正则匹配 | `body regex 'rO0AB'` |
| `gte` | 大于等于 | `body_entropy >= 7.0` |
| `subdomain_entropy` | 子域名熵值 | `>= 3.5` |
| `subdomain_length` | 子域名长度 | `>= 30` |

**偶尔需要修改**：新增条件操作符类型时。

---

### 3.6 `core/extractor.py` — IOC 提取（311 行）

**职责**：从流量中自动提取威胁情报指标（IOC）

**类**：
- `IOC` — 单条 IOC 记录（type, value, source, first_seen, last_seen, count, suspicious）
- `IOCExtractor` — 提取引擎
  - `extract(packet) -> list[IOC]` — 提取所有 IOC
  - `summary()` — 返回去重汇总
  - `export_iocs(path, data)` — 导出 JSON

**提取的 IOC 类型**：
| 类型 | 来源 |
|------|------|
| `ip` | 源/目的 IP + payload 中的 IP |
| `domain` | HTTP Host / DNS 查询 / TLS SNI |
| `url` | HTTP 请求完整 URL |
| `ua` | 非浏览器 User-Agent（标记可疑） |
| `ja3/ja3s/ja4/ja4s` | TLS 指纹 |

**一般不需要修改**。

### 3.7 `core/suspicion.py` — 可疑流量检测 + 导出（340 行）

**职责**：检测不构成确认告警但值得关注的行为，导出含 Wireshark 检索词的文本文件

**类**：
- `SuspiciousItem` — 可疑记录（含 wireshark_filter 和 grep_keywords 字段）
- `SuspicionEngine` — 检测引擎（逻辑同 AttackDetector，加载 suspicion.yaml）
- `SuspiciousExporter` — 文本导出器
  - `open()` — 打开文件，写入表头
  - `write(item)` — 写入一条记录
  - `close()` — 关闭文件，写入统计

**导出格式**（每行一条）：
```
时间 | 严重度 | src:port -> dst:port | 协议 | 指标摘要 | Wireshark过滤器 | grep关键字
```

**内置判断逻辑**：
- `not_browser_ua` — 排除常见浏览器 UA，剩余标记可疑
- `not_in` — 方法不在白名单中（PUT/DELETE/OPTIONS/TRACE）
- `in_list` — 端口在可疑列表中（8080/8443/8888 等）
- `range` — 数值范围匹配（熵值 5.0-7.0、子域名长度 20-30）
- `hour_range` — 时间范围（0:00-6:00 的 SSH/RDP 连接）
- `is_private` / `is_public` — 内网/外网 IP 判断

**偶尔需要修改**：新增操作符类型时。

### 3.10 `rules/signatures.yaml` — 确认攻击规则（18 条）

**⭐ 常改文件** — 新增/修改攻击检测规则

**当前规则覆盖 9 大类**：

| 类别 | 规则数 | 检测内容 |
|------|--------|---------|
| `sql_injection` | 4 | UNION SELECT / 时间盲注 / 布尔盲注 / 报错注入 |
| `cmd_injection` | 2 | Linux 命令注入 / Windows 命令注入 |
| `dir_traversal` | 1 | 目录遍历 / php://filter / file:// |
| `file_upload` | 1 | 可疑文件上传（.php/.jsp/.asp） |
| `webshell` | 3 | 冰蝎通信（熵值+对齐） / 哥斯拉 Cookie / 蚁剑 UA |
| `c2` | 3 | Cobalt Strike URI / DNS 隧道 / DNS TXT 外传 |
| `deserialization` | 1 | Java/PHP 反序列化 |
| `log4shell` | 1 | CVE-2021-44228 |
| `xss` | 1 | 跨站脚本 |

**每条规则结构**：
```yaml
- id: SQLI-001                    # 唯一标识
  name: "SQL Injection - UNION"   # 规则名称
  category: sql_injection         # 所属类别
  severity: high                  # 严重程度：critical / high / medium / low
  protocol: http                  # 协议过滤
  mode: pattern                   # 检测模式：pattern 或 condition
  target_fields: [uri, body]      # 检测字段（pattern 模式）
  patterns:                       # 正则列表（任一命中即触发）
    - '(?i)union\s+(all\s+)?select\s'
```

```bash

py main.py -i "以太网" --src-ip 10.0.0.50

py main.py -i "以太网" --suspicious-export --extract-ioc --duration 3600
```

## 五、三分类处理流程

```
数据包流入
    │
    ▼
┌──────────────────┐
│  攻击检测         │  ← rules/signatures.yaml
│  core/detector.py │
└────────┬─────────┘
         │
     命中？────是────▶  🔴 红色告警输出（终端 + 报告）
         │否               不再进入可疑检测
         ▼
┌──────────────────┐
│  可疑流量检测     │  ← rules/suspicion.yaml
│  core/suspicion.py│
└────────┬─────────┘
         │
     命中？────是────▶  🟡 黄色提示（终端） + 导出文本文件
         │否               文件含 Wireshark 过滤器和 grep 关键字
         ▼
     ⚪ 普通流量（仅计数，不输出）
```

**设计意图**：
- 确认攻击 = 误报率极低，可以立即响应（封 IP、隔离主机）
- 可疑流量 = 可能是攻击前兆或低置信度信号，需要人工审查
- 导出文件可直接用于 Wireshark/tshark 回溯分析

## 六、规则修改指南

### 6.1 新增确认攻击规则

编辑 `rules/signatures.yaml`，在 `rules:` 列表末尾追加：

```yaml
  # ============== 你的新规则 ==============
  - id: CUSTOM-001                         # 唯一 ID
    name: "Your Custom Detection"          # 规则名称
    category: webshell                     # 类别（可自定义）
    severity: critical                     # critical / high / medium / low
    protocol: http                         # 协议
    mode: pattern                          # pattern（正则）或 condition（条件）

    # pattern 模式：
    target_fields: [uri, body, cookie]     # 检测哪些字段
    patterns:
      - '你的正则表达式1'
      - '你的正则表达式2'

    # 或 condition 模式：
    conditions:
      - field: method                      # 字段名
        operator: eq                       # 操作符
        value: "POST"                      # 期望值
      - field: body_entropy
        operator: gte
        value: 7.0
```

### 6.2 新增可疑行为规则

编辑 `rules/suspicion.yaml`，格式同上，但 `severity` 用 `high_suspicion` / `medium_suspicion` / `low_suspicion`。

可新增的操作符：

| 操作符 | 说明 | value 示例 |
|--------|------|-----------|
| `eq` | 等于 | `"POST"` |
| `contains` | 包含 | `"octet-stream"` |
| `regex` | 正则匹配 | `'\.php(\?\|$)'` |
| `not_browser_ua` | 非浏览器 UA | `true` |
| `not_in` | 不在列表中 | `"GET,POST,HEAD"` |
| `in_list` | 在列表中 | `"8080,8443,8888"` |
| `range` | 数值范围 | `"5.0-7.0"` |
| `hour_range` | 小时范围 | `"0-6"` |
| `is_private` | 内网 IP | `true` |
| `is_public` | 公网 IP | `true` |

### 6.3 新增可疑工具 UA

编辑 `rules/ioc_patterns.yaml`，在 `suspicious_patterns` 列表中追加：

```yaml
  user_agent:
    suspicious_patterns:
      - '(?i)你的工具名'
      # ... 现有条目 ...
```

同时更新 `core/suspicion.py` 中的 `TOOL_UA_PATTERNS` 列表（两处需同步）。

### 场景五：用导出文件回溯分析

```bash
# 导出的可疑流量文件中每行包含 Wireshark 过滤器，例如：
# ip.addr==10.0.0.51 && tcp.port==8080 && http.request.uri contains "/actuator"

# 在 Wireshark 中直接粘贴该过滤器即可定位到对应数据包
# 或用 tshark 命令行：
tshark -r capture.pcap -Y "ip.addr==10.0.0.51 && tcp.port==8080 && http.request.uri contains /actuator"
```
